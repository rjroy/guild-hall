MEETING NOTES: Heartbeat Commission Dispatch Spec Review
Meeting with Octavia, 2026-04-03

DISCUSSION SUMMARY

The user identified a gap in the heartbeat error handling specification. While REQ-HBT-6 outlined per-project failure handling, it did not address loop-level behavior when rate limits are encountered. Since rate limiting is the most likely failure mode for subscription users, the specification needed explicit guidance on backoff strategy.

The user clarified that rate limiting is not a transient error to retry around—it indicates the query has already failed and further project evaluations in the same tick would burn requests into the same wall. Therefore, when a rate limit error occurs, the entire tick should abort and the heartbeat loop should wait before attempting the next tick.

The user chose to make the backoff duration configurable rather than hardcoding it. While Anthropic's 5-hour subscription window is unlikely to change, a configuration parameter provides flexibility for hypothetical future scenarios and aligns with the principle of making externalized constraints tunable.

KEY DECISIONS AND REASONING

Rate-limit errors abort the tick. When any project's heartbeat session fails with a rate-limit error, the loop stops evaluating remaining projects and schedules the next tick after the backoff interval. Other errors (SDK failures, timeouts) continue per-project; only rate limits trigger loop-wide backoff. Rationale: A rate limit means the query failed; continuing to evaluate remaining projects wastes requests against the same limit.

Backoff duration is configurable with a 5-hour default. Added REQ-HBT-28a: `heartbeatBackoffMinutes` in config, defaulting to 300 minutes (5 hours), with a minimum of 60 minutes. This reflects the subscription rate-limit window while allowing users to override if needed. Rationale: Anthropic's billing cycle is unlikely to change, but externalized configuration prevents spec rewrites if it does.
