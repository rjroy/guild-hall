**Meeting: Close bootstrap gap in read-only verification tools spec**

The session identified and resolved a load-bearing gap in the read-only verification tools spec. The verification toolbox requires a `.lore/guild-hall-config.yaml` file to function, but the spec provided no mechanism for that file to come into existence. The motivating worker (Thorne, the reviewer) has no Write capability and cannot create the file, meaning the tools would ship non-functional without intervention.

The discussion surface identified four missing pieces: when the config file is created, how the gap is signaled, how it gets resolved, and where it's recorded. The user decided on complete automation at project registration time, with empty projects still receiving the template as a persistent record of the gap.

Five new requirements were added to the spec (REQ-VFY-22 through REQ-VFY-26) implementing the bootstrap loop: project registration automatically writes a template `.lore/guild-hall-config.yaml` with four empty `checks` keys and files a tracking issue to `.lore/issues/`. Empty string values are treated identically to missing keys, allowing the template to ship inert and projects to mark checks as "not applicable." For projects registered before this spec landed, daemon startup performs the same reconciliation on each registered project, writing the template and filing the issue where missing. Both registration-time and startup-time writes are committed to the `claude` integration branch with message `"chore: bootstrap verification config"`, ensuring subsequent activity worktrees inherit the files.

**Artifacts:** specs/infrastructure/read-only-verification-tools.md (status updated from draft to approved)

**Next step:** `/lore-development:prep-plan` to build the implementation plan. The work touches three daemon surfaces (toolbox factory registration, registration handler extension, startup reconciliation) and requires a new config-parsing library.

## Decisions

**How should `.lore/guild-hall-config.yaml` be created in projects so the verification toolbox is usable without manual user setup?**
Project registration writes a template config with four empty `checks` keys and files a tracking issue on the same transaction. Empty string values behave identically to missing keys. Daemon startup reconciles registered projects that predate the spec, writing the template and filing the issue for any project whose integration worktree lacks the config file. Both writes are committed to the `claude` integration branch as `chore: bootstrap verification config`.
*Reasoning: The motivating worker (Thorne) has no Write capability and cannot bootstrap the file itself. Without automatic creation, the tools would ship dead on arrival — the "not configured" message in REQ-VFY-14 would land in worker output where the gap may never get noticed or closed. The user confirmed full automation is desired, and that empty projects should still get the template as a holding record. Startup reconciliation handles projects registered before this spec landed without requiring user intervention.*
