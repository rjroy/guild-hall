MEETING NOTES — Guild Hall Audience with Dalton
March 10, 2026

SUMMARY

The session focused on two enhancements to the memory injection system in the Guild Hall daemon. The first item was adding a persistent guidance preamble to the `loadMemories` function in `daemon/services/memory-injector.ts`. Previously, `loadMemories` returned an empty string when no memory files existed, meaning workers never received any instruction about the `write_memory` tool. The change introduced a `MEMORY_GUIDANCE` constant explaining the tool, its three scopes (global, project, worker), and what is worth saving versus skipping. This guidance now appears in every `memoryBlock` response regardless of whether files exist, and is excluded from the character budget so it does not consume memory allocation reserved for file content.

The second item was increasing `DEFAULT_MEMORY_LIMIT` from 8,000 to 16,000 characters, prompted by Dalton's assessment that 8,000 was too restrictive. The change was applied to the constant itself and to all associated JSDoc comments referencing the old value. Both changes were scoped to `loadMemories` so the manager context path, which calls `loadMemories` via an injected dependency, picks up the behavior automatically without separate changes.

After the limit increase, four tests in `tests/daemon/memory-injection.test.ts` failed because they asserted `memoryBlock` equaled an empty string in over-budget and no-files scenarios. The tests were updated to assert against the new guidance-containing strings and the compaction message. The `memoryLimit: 8000` reference in one test was also updated to 16000. All 19 tests in the file passed after the fixes. Stub usages of `loadMemoriesFn` in other test files were confirmed to be intentional fake returns unrelated to the real implementation and required no changes.

KEY DECISIONS

DEFAULT_MEMORY_LIMIT doubled to 16,000 characters. Rationale: Dalton judged 8,000 to be too small for practical use.

Guidance preamble excluded from the file content budget. Rationale: keeping it inside the budget would silently reduce the space available for actual memory files, defeating the purpose of the limit setting.

Guidance injected via `loadMemories` rather than in worker activation or system prompt assembly. Rationale: ensures the manager context path receives it as well, since both paths consume `loadMemories`.

ARTIFACTS PRODUCED

daemon/services/memory-injector.ts — modified to add `MEMORY_GUIDANCE` constant and update `loadMemories` to always return a non-empty `memoryBlock`; `DEFAULT_MEMORY_LIMIT` raised from 8,000 to 16,000.

tests/daemon/memory-injection.test.ts — four tests updated to reflect new non-empty return values in empty and over-budget cases; one test updated from `memoryLimit: 8000` to `memoryLimit: 16000`.

OPEN ITEMS

None explicitly raised. Dalton did not request test updates himself; the assistant proposed and completed them. No follow-ups were deferred.
