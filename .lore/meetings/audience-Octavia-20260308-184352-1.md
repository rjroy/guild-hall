Meeting — Audience with Octavia, Guild Chronicler
Date: 2026-03-08

Octavia conducted a structured review of the model-selection spec (.lore/specs/infrastructure/model-selection.md), cross-referencing its claims against the live codebase. The review examined the spec through four lenses: clarity, completeness, consistency, and verifiability. Her methodology involved reading the spec, locating every code path the spec describes or implies, and comparing actual behavior against stated requirements. Relevant files examined included the briefing generator, the shared WorkerMetadata type, the commission orchestrator, the session preparation pipeline, the manager toolbox, and both the scheduled commissions and commission specs that model-selection references.

Three important findings drove the substance of the session. First, REQ-MODEL-13 was identified as misleading: it describes the briefing generator as already using the resource_overrides path, when in fact it applies a direct spread on the prepared options object. The requirement reads as a description of existing state rather than a refactoring directive, which would confuse any implementer who looks at the actual code. Second, REQ-MODEL-10 relies on REQ-SCOM-11 from the scheduled commissions spec to carry model through to spawned commissions, but that spec never mentions model — only maxTurns and maxBudgetUsd — making the handoff a gap rather than a handled case. Third, the Overview's statement that briefings use the worker's default model conflicts with the briefing generator's existing Sonnet hardcode and with what REQ-MODEL-13 requires, leaving the intended end state ambiguous.

Additional findings included: REQ-MODEL-2's "same way as roster workers" is ambiguous because the Guild Master is constructed programmatically rather than from a package.json; REQ-MODEL-7 does not flag that REQ-COM-2 in the commissions spec requires amendment to include model in resource_overrides; the create_commission tool's parameter extension is implied but never explicitly required; and REQ-MODEL-16 through REQ-MODEL-18 use "displayed" without specifying location, visibility conditions, or format, referencing view specs that do not yet mention model display at all.

Key decisions made: The spec was updated in five targeted edits to address the priority findings. REQ-MODEL-13 was rewritten to separate current state from the refactoring requirement. REQ-MODEL-10 was amended to explicitly call out that the scheduled commissions spec needs updates to REQ-SCOM-11 and REQ-SCOM-19. The Overview inconsistency regarding briefings was resolved. The remaining minor findings (REQ-MODEL-2 ambiguity, commission spec gap, create_commission tool parameter gap, verifiability of display requirements) were noted in the review document but not all required immediate spec edits — those represent follow-on work.

Artifacts produced or referenced: The model-selection spec (.lore/specs/infrastructure/model-selection.md) was the primary artifact reviewed and edited. Referenced artifacts include .lore/brainstorm/model-selection.md (prior brainstorm), .lore/specs/commissions/guild-hall-scheduled-commissions.md, .lore/specs/commissions/guild-hall-commissions.md, daemon/services/briefing-generator.ts, packages/shared/worker-activation.ts, daemon/services/manager/worker.ts, daemon/services/manager/toolbox.ts, and lib/types.ts.

Open items: The scheduled commissions spec (guild-hall-scheduled-commissions.md) requires amendment to REQ-SCOM-11 and REQ-SCOM-19 to add model to resource_overrides. The commissions spec (guild-hall-commissions.md) requires amendment to REQ-COM-2 for the same reason. The view specs (guild-hall-views.md and guild-hall-worker-roster.md) require model display requirements to be added before REQ-MODEL-16 through REQ-MODEL-18 are verifiable. The Guild Master's model declaration path (REQ-MODEL-2) remains under-specified and should be clarified before implementation begins.
