**DISCUSSION SUMMARY**

Octavia presented Verity's research on four P4 isolation approaches that map to Guild Hall's git-based worktree model. The research evaluates workspace-based isolation, task streams, git-p4 bridging, and shelf-only workflows against Guild Hall's requirements for branch isolation, sparse checkout, and integrated merge flows. After review, the user proposed a fundamentally different fifth approach: treating P4 as a filesystem source of truth with a disposable local git layer, rather than attempting to make P4 itself provide isolation.

The core insight of the proposed approach is that Guild Hall's VCS-layer code need not change at all. A simple bootstrap-and-teardown wrapper handles the P4 bookends: `p4 sync` before work, git integration and shelving after, then reset and repeat. The git repo remains real throughout the commission lifecycle—all isolation, branching, and three-tier merging work unchanged. This avoids the P4 server load problem that invalidates all four research approaches: those approaches add server traffic during work (workspace syncs, stream operations, shelf manipulation), whereas the proposed approach limits server interaction to initial sync and final shelve.

The user clarified Guild Hall's actual operational constraints that reshape the analysis: commissions are atomic (one PR, squashed), all commissions must resolve before PR merge (no active worktrees at reset), and the depot is 100GB+ with 100M+ commits, making git-p4 clone prohibitively slow. Under these constraints, the proposed approach is the only one that keeps server load minimal.

---

**KEY DECISIONS & REASONING**

1. **Document as brainstorm, not research extension.** The research stands as Verity wrote it. This is an outcome of reading the research, not a correction to it. Brainstorm format is appropriate for exploratory ideas.

2. **Scope the "guild hall spec" for binary handling.** Rather than track all P4 content in git, restrict git to specific directories (source code directories only). This keeps `.git` small, avoids binary bloat, and enforces that commissions only touch code-relevant paths. Aligns with Guild Hall's sparse checkout principle.

3. **Accept edge cases 4 and 6 as non-issues.** Active worktrees at reset time and multiple shelves per cycle are already precluded by Guild Hall's operational model (atomic PRs, all commissions resolved before merge). No design work needed.

4. **Defer edge case 1 (drift during work) to separate brainstorm.** The P4 sync-based rebase flow is messy but viable. This deserves its own focused exploration with examples and conflict scenarios.

**OPEN ITEMS**

1. **Drift handling during commission work.** Define the integrate-back flow when P4 has moved ahead during git work. Identify what conflict scenarios surface and how reviewer handles them. (Candidate for separate brainstorm: "P4 sync-based rebase flow")

2. **Guild Hall spec definition.** Specify which directories are tracked by git, which are ignored. Tie to project's actual source structure.

3. **Integration-back script reliability.** The P4 adapter must translate git changes to explicit `p4 add/edit/delete` commands. Verify behavior on file renames, deeply nested directories, and mixed additions/deletions.

4. **P4 metadata coexistence.** Confirm `.p4ignore` excludes `.git/` and `.gitignore` excludes P4 metadata. Test in actual studio P4 setup.

5. **Wrapper script as external tool.** Explore whether the P4 adapter can live entirely outside Guild Hall codebase—a deployment-time wrapper rather than daemon integration. If viable, keeps Guild Hall itself VCS-agnostic.
