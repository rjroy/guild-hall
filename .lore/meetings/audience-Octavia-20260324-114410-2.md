MEETING NOTES: Windows Native Support Brainstorm Review
Guild Hall | 2026-03-24 | Octavia

SESSION SUMMARY

Octavia presented a comprehensive brainstorm document on enabling Guild Hall to run natively on Windows, covering socket dependencies, IPC alternatives, Bun compatibility, non-socket POSIX assumptions, and a four-option architecture approach. The session focused on resolving five open questions that would shape the implementation strategy. The user and Octavia worked through each question systematically, making decisions that tighten the brainstorm's scope and confirm the recommended approach.

DECISIONS AND REASONING

1. Don't rely on Bun Unix socket support on Windows. Proceed with TCP fallback design regardless of future Bun AF_UNIX additions. Reasoning: Windows AF_UNIX is a 2018 kernel feature Microsoft hasn't invested in deeply. Building on it through Bun's compat layer represents two layers of uncertainty. TCP localhost is a proven, native Windows primitive.

2. Strike Question 2 (node:http socketPath behavior on Windows) from the brainstorm. Reasoning: TCP fallback makes this question irrelevant. The client will connect via hostname:port, not socketPath. This question was only pertinent if Windows AF_UNIX remained a candidate.

3. Confirm TCP security is acceptable. Bind explicitly to 127.0.0.1 (not the string "localhost") to avoid IPv4/IPv6 resolution ambiguity on Windows. Document the local-process visibility trade-off. Reasoning: Guild Hall is a personal development tool. TCP loopback is not network-accessible. Any attacker with local code execution already has access to filesystem, repos, and keys. The daemon is not the threat model's weakest link.

4. Auto-detect transport by platform (win32 gets TCP, all others get Unix socket). Do not add a configurable transport setting to config.yaml. Reasoning: Platform detection is deterministic and requires zero user decision-making. No realistic use case exists for override (WSL and native Windows on same machine maintain separate Guild Hall installations). Configuration knob adds documentation and testing burden with no practical benefit.

5. Treat Claude Agent SDK sandbox behavior on Windows as an external dependency risk, not a Guild Hall architecture concern. Document and move on. Reasoning: Guild Hall delegates sandboxing to the SDK. The existing bubblewrap check is already platform-gated. The SDK owns per-platform sandbox implementation.

ARCHITECTURE IMPLICATIONS

Options C (Windows AF_UNIX) and 4 (Node.js HTTP server with AF_UNIX) are eliminated from consideration. Option 2 (TCP fallback on Windows only) is confirmed as the implementation path. The comparison matrix has been updated to remove AF_UNIX. The four migration phases remain valid: Phase 0 (non-socket portability fixes), Phase 1 (transport abstraction for platform-aware daemon binding), Phase 2 (verification and hardening), Phase 3 (polish and documentation). Phase 3 no longer includes questions about config settings.

ARTIFACTS

.lore/brainstorm/windows-native-support.md was updated to reflect all five resolutions. Eliminated options now include reasoning. Open Questions section replaced with Resolved Questions. Recommendation confirmed (not tentative).

FOLLOW-UP

User is closing this session. Specification work for Windows native support will proceed in a fresh session. The brainstorm is ready for architectural design and implementation planning.

## Decisions

**Should we wait to see if Bun adds Unix socket support on Windows before designing the transport abstraction?**
No. Proceed with TCP fallback design regardless of Bun's future AF_UNIX support on Windows.
*Reasoning: Windows AF_UNIX is a bolt-on, not a native primitive. Even if Bun wires it up, building on a flimsy OS integration through a runtime's compat layer is two layers of uncertainty. TCP localhost is proven and native on Windows. Design for what's solid.*

**Is the node:http socketPath behavior on Windows still a relevant open question?**
No. Strike Question 2 from the brainstorm.
*Reasoning: With TCP fallback as the Windows transport, the client connects via hostname:port, not socketPath. The question was only relevant if AF_UNIX was a candidate path, and it's not.*

**Is TCP localhost security acceptable for Guild Hall on Windows?**
Yes. Bind to 127.0.0.1 explicitly, document the local-process visibility trade-off, move on.
*Reasoning: Guild Hall is a personal development tool. TCP on loopback is not network-accessible. Any attacker with local code execution already has access to the filesystem, git repos, and SSH keys. The daemon is not the weakest link in that threat model. Binding to 127.0.0.1 (not "localhost" string) avoids IPv4/IPv6 resolution ambiguity on Windows.*

**Should the transport type (Unix socket vs TCP) be a configurable setting in config.yaml?**
No. Auto-detect by platform. win32 gets TCP, everything else gets Unix socket.
*Reasoning: The user's framing: "as long as it's a no brainer to decide *nix or not, then auto detect." Platform detection is deterministic and trivial. There's no realistic scenario where a user needs to override it. A config knob adds documentation, testing, and error-handling burden for zero practical benefit.*

**Is the Claude Agent SDK's Windows sandbox behavior a Guild Hall design concern?**
No. It's an external dependency risk, not a Guild Hall architecture question. Document it and move on.
*Reasoning: Guild Hall delegates sandboxing to the SDK. The bubblewrap check is already platform-gated (Linux only). The existing sandbox work accidentally supports this by not assuming any particular sandbox mechanism. If the SDK's Windows sandbox has issues, that's the SDK's problem to solve.*
