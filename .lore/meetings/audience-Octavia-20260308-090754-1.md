Meeting held 2026-03-08. Participants: user and Octavia (Guild Hall meeting agent).

The meeting focused on the design of scheduled commissions — a mechanism for the Guild Hall daemon to fire recurring commissions on a defined cadence. The user provided clarifications on five open design questions from a prior discussion: schedules should be defined within the commission artifact itself (stored in the project repo), the artifact should record the last run timestamp so the daemon can determine catch-up state on startup, a commission that is still running should not be re-triggered (and if stuck past a reasonable threshold, the Guild Master should escalate to the user via a meeting request), the "assuming yes" behavior is a usage concern not a system design concern, and the Guild Master must be able to create and edit schedules programmatically via toolbox tools.

From those inputs, the design was fleshed out in detail. Scheduled commissions introduce a new type field (one-shot vs. scheduled) rather than overloading the existing status enum. The status field for scheduled commissions tracks active, completed, and failed, with paused left as a possible future addition. Each scheduled commission artifact serves dual purpose: it holds the standing schedule definition (cron expression, worker, prompt, dependencies) and accumulates spawn records in its activity_timeline. When the schedule fires, a normal one-shot commission is created with a source_schedule back-reference; that spawned commission has its own full lifecycle, worktree, and artifact. The cron format was chosen over human-readable intervals or ISO 8601 durations because schedule editing is primarily done by an LLM via a tool, not by humans editing files. On log growth, the preferred approach is to let the schedule artifact timeline grow and rely on a periodic compression pass (consistent with the existing tend pattern) to summarize old entries rather than a rolling window or a separate log file.

Key decisions: cron expressions for cadence, type: one-shot vs. scheduled as a first-class field, active/completed/failed status model for scheduled commissions, tend-based compression for timeline growth, Guild Master gets create_scheduled_commission and update_schedule tools. The brainstorm was saved to .lore/brainstorm/scheduled-commissions.md and linked to the meeting artifact.

Open items for spec work: exact schema for the schedule block in the commission artifact frontmatter, definition of the "stuck" threshold that triggers escalation, cron evaluation library selection for Bun, and formal specification of the two new Guild Master toolbox tools.
