# Guild Hall Meeting Notes: Octavia
**Date**: 2026-03-17 | **Audience**: Octavia

## Summary

The meeting addressed a critical architectural decision about Guild Hall's capability model: whether agents and humans share a single "skill" surface (convergence model) or use separate mechanisms. The team reviewed a Daemon Application Boundary spec revision and identified gaps in an accompanying rename plan that would deprecate "skill" terminology in the codebase and reserve it for Claude Code skills only. Discussion focused on ensuring the rename plan accounted for all affected layers—not just TypeScript types caught by the compiler, but also CLI-local types, wire format changes, and runtime string literals in tool descriptions.

The team confirmed that the convergence model was never fully implemented and that agent interaction already routes through MCP tools via the daemon's toolbox system. The half-built skill injection infrastructure (formatSkillDiscoveryContext, SessionPrepDeps.skillRegistry) serves no active purpose and should be removed. The name "skill" was causing confusion between daemon REST/CLI capabilities and Claude Code plugin skills, which are separate concepts. The decision to rename the former to "operations" provides clear terminology and unblocks the rename plan.

Three artifacts emerged from this work: a revised DAB spec establishing agents use MCP tools and humans use REST/CLI operations, a 12-step rename plan covering the full blast radius including CLI layer and wire format, and an architectural issue documenting that the operations-loader should derive from toolboxFactory rather than require a separate factory export.

## Key Decisions

**Decision: Agent-daemon interaction model**
- Agents interact with Guild Hall through MCP tools provided by the daemon's toolbox system. The skill-convergence model (agents and humans sharing a single REST/CLI surface) is abandoned.
- Reasoning: The convergence model was never fully implemented. Agents still use MCP tools, which work fine. The skill injection infrastructure was unused noise. Guild Master receives CLI access as a special case (additive to toolbox MCP tools).

**Decision: Terminology**
- The term "skill" is reserved for Claude Code skills (plugin artifacts). Daemon REST/CLI capabilities are called "operations." This distinction is now explicit in the DAB spec and governs the codebase rename.
- Reasoning: "Skill" was causing confusion between three distinct concepts: operations (daemon REST/CLI), MCP tools (agent interaction), and Claude Code skills (plugins). Clear terminology prevents architectural mistakes.

**Decision: Scope of rename**
- The rename touches five layers: core types, agent injection removal, route factories, CLI layer, and wire format. The plan includes grep-based verification for string literals and CLI-local types that typecheck will miss.
- Reasoning: Type renames are caught by the compiler, but the CLI layer defines CliSkill locally (not imported), and manager toolbox contains [skillId: ...] annotations in strings. Mechanical rename alone would leave stale terminology.

- Revised architecture: agents use MCP tools, humans use operations, Guild Master gets CLI as special case.
- Retired REQ-DAB-7 through -15 (skill-convergence model). New requirements REQ-DAB-16 through -23.

**Plan**: `.lore/plans/infrastructure/skill-to-operations-rename.md`
- 12-step mechanical rename covering daemon, lib, cli, and tests.
- Explicitly addresses three categories of missed references: TypeScript identifiers (caught by typecheck), CLI local types (caught by grep), and string literals (caught by targeted grep).
- Provides file lists and verification guidance including a sub-agent spot-check of the CLI layer.

**Issue**: `.lore/issues/operations-loader-should-derive-from-toolbox-factory.md`
- Operations loader should read toolboxFactory and derive operation definitions from it, rather than require a separate operationFactory export.
- Currently low-impact (no packages export operationFactory). Should be addressed when a package first contributes REST/CLI operations.

## Open Items and Follow-ups

The three artifacts are self-contained and ready for implementation. An implementer can follow the rename plan from the spec and issue alone without revisiting this chat. The 12-step plan includes explicit verification steps (typecheck, lint, tests, and a grep pass) and delegates the lore docs review to a fresh-context reviewer, who should distinguish between "skill" meaning daemon operation (rename) vs "skill" meaning Claude Code plugin (keep unchanged).
