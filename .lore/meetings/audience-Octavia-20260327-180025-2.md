GUILD HALL MEETING NOTES
2026-03-28 | Audience: Octavia | Topic: Guild Master Capability Discovery

SUMMARY
Octavia and the Guild Master investigated how the manager learns about worker capabilities at activation time. The investigation traced the complete system: worker metadata declaration in package.json (identity, posture, toolboxes, built-in tools, domain plugins), the package discovery pipeline, and the manager context builder that formats this information into markdown and injects it into the manager's system prompt. The manager's awareness is assembled from seven layers: soul, identity, posture, model guidance, injected memory, meeting/commission context, and operational state. A capability gap was identified: domain plugins (which contain skills like /cleanup-commissions) are not exposed in the manager's global awareness section, only resolved per-worker during individual SDK session preparation.

DISCOVERIES
The manager learns worker capabilities through static metadata (from package.json's guildHall key) combined with dynamic context assembly at activation time. Worker metadata includes identity, posture, domain toolboxes, built-in tools, and optional domain plugins. The buildManagerContext() function in daemon/services/manager/context.ts iterates discovered packages and extracts worker metadata, formatting it into a markdown worker section that lists name, description, checkout scope, toolboxes, and tools. This context is injected as part of the manager's system prompt during activation. Domain plugins are resolved during prepareSdkSession() but not surfaced in the manager's static awareness—the manager knows workers have plugins but not which skills those plugins provide.

FOLLOW-UPS
