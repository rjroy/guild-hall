The first finding identified misleading `app.request()` phrasing in REQ-26 and the AI Validation section—`app.request()` is Hono's HTTP test client, but once the help routes are unregistered there's no REST surface to hit; the actual seam is the DI factory. The second finding noted that success criteria line 154 (commission paths change) doesn't pin to a single REQ but is instead a composite criterion. All 28 requirements now trace cleanly to ownership boundaries, requirements, success criteria, or constraints with no orphans. Spec is self-consistent and ready to anchor the next plan revision.

## Decisions

**Does every Success Criteria bullet trace to a REQ?**
Seventeen of eighteen SC bullets trace cleanly. One bullet ("Commission operations remain invocable... their CLI paths are updated to the new surface") and one bullet ("Package-contributed operations are either (a) mapped... (b) fallback... (c) excluded") are not pinned to a single REQ — they trace to Constraints rather than Requirements. Minor, because both are scoped as planning-decision criteria, which is what Success Criteria can legitimately record, but a reader looking for a direct REQ source will not find one.
*Reasoning: Walked every SC bullet against the 28 numbered requirements. Bullet on commission paths updating to the new surface has no explicit REQ saying "commission paths must change" — it's implied by REQ-5 (task-oriented top level applies to every noun, including commission). Bullet on package-op policy is explicitly deferred to a follow-up spec in Constraints and Exit Points.*

**Does every AI Validation item trace to a REQ?**
Yes. All nine AI Validation items map to one or more REQs with no orphans.
*Reasoning: Path-rule tests → REQ-8/9/12/13/21. Help-completeness → REQ-16/17/18. Skill-build → REQ-19/20. CLI mapping ↔ catalog → REQ-3/4/22. Daemon help surface removal → REQ-26. Daemon leaf presence → REQ-22. No-cliPath → REQ-2. Formatter-keying → REQ-25. Meeting list aggregation → REQ-10a.*

**Is the "app.request() pattern" phrasing accurate in REQ-26 and the AI Validation consistency test?**
Slightly imprecise. app.request() is Hono's in-process HTTP test client — it exercises the mounted REST surface, not the injected OperationsRegistry directly. With the help routes unregistered, app.request() cannot reach the registry at all; the only path is via a DI seam. The intended meaning appears to be "the same in-process test harness that wires up deps," but naming app.request() specifically confuses the mechanism.
*Reasoning: Reviewed line 58 (REQ-26) and line 171 (AI Validation). Both cite "app.request() pattern" as the way to access OperationsRegistry. This conflates the HTTP test client with the factory-injected registry. A reader implementing the test would be misled into thinking there's a REST call involved.*

**Is REQ-26 in the right section (Ownership Boundary)?**
Defensible but not ideal. "Ownership Boundary" currently contains REQs that establish who owns what (CLI owns surface, daemon owns capability catalog, CLI is a thin client, daemon grammar is unchanged). REQ-26 is a teardown directive, not an ownership assertion. It fits by association (the CLI no longer consults daemon help routes, so they are no longer shared surface) but a "Scope Teardown" or "Daemon Surface Changes" subsection would be a cleaner home. Since the section label is "Ownership Boundary" and REQ-26 decisively ends the daemon's ownership of help-surface, keeping it here is acceptable.
*Reasoning: Other REQs in the section are statements of current ownership. REQ-26 is an action — remove routes. Sibling REQ-4 has been tightened to cross-reference REQ-26, which means they are meant to be read together. Moving REQ-26 would break that proximity. Net: keep where it is.*

**Does the spec contain any stale "removing cliPath" phrasing after G1 edits?**
No. All five cliPath mentions (Overview line 34, REQ-2 line 52, Success Criteria line 140, AI Validation line 177, Context line 196) describe cliPath as absent and staying absent, with a compile-time guard. The Context paragraph correctly records that the field was proposed in the 2026-03-14 cli-rewrite plan for SkillDefinition but never shipped.
*Reasoning: Verified against lib/types.ts (grep found no cliPath references) and against .lore/plans/infrastructure/cli-rewrite-thin-daemon-client.md Phase 1, which proposed adding cliPath? to SkillDefinition but was never executed (subsequently superseded by the skill-to-operations rename).*

**Is REQ-CLI-AGENT-26 unambiguous about scope, consumers, and replacements?**
Mostly. Enumerates all six routes that actually exist in daemon/routes/help.ts. Names CLI as primary consumer and the test catalog-consistency check as the secondary consumer, with replacements for each. Does not overreach into implementation. Does not contradict REQ-4.
*Reasoning: Verified route list via grep of daemon/routes/help.ts: GET /help, GET /help/operations, GET /:root/help, GET /:root/:feature/help, GET /:root/:feature/:object/help, GET /:root/:feature/:object/:operation/help. All six appear in REQ-26. REQ-4 is scoped to capability operations and explicitly cross-references REQ-26 for help routes.*
