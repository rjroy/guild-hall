MEETING NOTES: Brainstorm: Commit .lore from web UI
Attendees: Guild Hall team with Octavia (chronicler)
Date: 2026-03-15
Duration: ~4 minutes

SUMMARY

The team reviewed a brainstorm document proposing a feature to allow users to commit .lore artifact changes directly from the web UI, removing the friction of going to the terminal. The document presented three design questions with clear recommendations: daemon route placement at /workspace/git/lore/commit in a new git-lore.ts file, UI placement as an action bar above the artifact tree matching the commissions tab pattern, and state awareness through a hasPendingChanges indicator on the button. All three design recommendations were accepted without modification.

The focus of the meeting was resolving four open questions left in the brainstorm. The user confirmed satisfaction with the overall approach and wanted to make explicit decisions on each question before moving to specification or implementation planning.

DECISIONS MADE

1. Push on commit: Decided NO. Commits remain local to the integration worktree. Users who want to push can trigger a sync from the admin panel. This keeps commit and push as separate operations with local-commit as the safer default.

2. Empty commit message handling: Decided to REJECT with HTTP 400. Require the user to provide a meaningful message. The purpose of this feature is deliberate, user-authored commits, and defaulting to a generic message would undermine that intent. UI validation should prevent submission with empty text.

3. Conflict with per-write auto-commits: No action needed. Confirmed that auto-commits from individual write operations and user-authored deliberate commits coexist naturally in the history, mirroring a developer's normal workflow of incremental saves plus deliberate checkpoints.

4. fileCount in status response: Decided to INCLUDE and SURFACE in the UI. The count should appear in the inline commit form (alongside the message input) not just as a button indicator. Seeing a file count that doesn't match user expectations provides a natural pause moment for verification before committing.
