MEETING NOTES: Test Output Noise Investigation
Guild Hall audience with Sable | 2026-03-13

SUMMARY

Investigated console output noise in the test suite that obscures actual test failures. Guild Hall has 236 console calls across 23 daemon files and 10 across 2 library files, with manual save/restore patterns in 5 test files. Two tests are currently failing but their failures are difficult to diagnose due to output noise from logging in production code.

Research identified three patterns used by mature Node.js projects: (1) framework-level suppression (Jest --silent, Vitest onConsoleLog, neither available in Bun), (2) structured loggers (pino/winston with log levels), and (3) injectable logger abstraction via dependency injection. Guild Hall already uses the DI callback pattern in sdk-logging.ts, establishing precedent for injectable logging.

Bun test offers no built-in console suppression. The preload script mechanism via bunfig.toml exists but is a blunt instrument that loses debugging information when tests fail. The injectable logger approach aligns with Guild Hall's existing architecture (DI used throughout services, toolbox factories, event bus) and would provide the real solution.

KEY DECISIONS

No action taken at this time. Deferred pending completion of other work in flight. Acknowledged that test output noise must be addressed to enable proper diagnosis of two known failing tests and prevent future test suite degradation.

ARTIFACTS PRODUCED

Created `.lore/issues/test-output-noise-from-raw-console.md` to track the problem and its two sub-issues: (1) raw console noise burying real failures, (2) two tests that fail in full suite but pass in isolation (likely cross-test state bleed exacerbated by noise).

OPEN ITEMS

Implement solution for test output suppression (deferred). Recommendation is Option 2 (injectable logger) as the real fix: create a Log interface with consoleLog default and nullLog for tests, thread through the 23 files currently using raw console calls. Option 1 (preload script in bunfig.toml) viable as immediate band-aid but loses debugging information. Option 3 (structured logger library) overengineered for a local dev tool. Diagnose the two test failures once noise is mitigated.
