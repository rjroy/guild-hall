MEETING NOTES: Commission List Filtering - Multi-Select Status Checkboxes

DISCUSSION

Octavia reviewed the commission list filtering brainstorm with four unresolved design questions. The feature addresses a usability gap: the commission list contains 80+ items with no way to filter to actionable work. The approach uses independent checkboxes for each status (not tabs), allowing users to see combinations like "in_progress + sleeping + failed" simultaneously. Default selection highlights items requiring attention without user interaction.

Four design decisions were required to move forward. The user approved recommendations that resolve all of them: `blocked` status defaults to on because PR #103 moved it into the "needs attention" artifact group, signaling it requires action rather than being deliberately parked. The checkbox layout groups by commission sort order (Idle, Active, Failed, Done) with 2-4 items per row and sort group labels for semantic clarity. A "Reset to defaults" toggle helps users return to the useful default set after toggling. Count display appears next to each checkbox only for statuses containing at least one commission.

DECISIONS

`blocked` status defaults to ON. Rationale: PR #103 reorganized `blocked` into the same artifact sort group as `failed` and `cancelled`, indicating the system treats it as needing attention rather than being deliberately parked. `paused` remains off by default.

Checkbox layout groups by sort group with row structure. Each group (Idle, Active, Failed, Done) renders as a compact horizontal row with 2-4 checkboxes using StatusBadge gem + label formatting. Group labels serve as both UI organization and documentation of how commissions sort in the list below.

"Reset to defaults" toggle included. Users can restore the useful default set of statuses in one click after toggling.

Count display shown conditionally. Checkboxes show commission counts (e.g., `In Progress (2)`) only for statuses with at least one commission to reduce noise.

IMPLEMENTATION NOTES

CommissionList will convert from server to client component using `"use client"` and `useState` for the selected status set. Filtering is purely client-side using `Set<string>.has(commission.status)` against the CommissionMeta[] props. No server changes required. URL parameter persistence and text search remain out of scope.
