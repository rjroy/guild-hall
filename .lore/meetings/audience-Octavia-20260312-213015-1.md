MEETING NOTES: Daemon Application Boundary Plan Review
Date: 2026-03-13 | Attendees: Octavia (Guild Chronicler), Audience

SUMMARY

Octavia presented the "Daemon Application Boundary Migration" plan, a draft document outlining an eight-phase strategy to migrate Guild Hall from its current architecture (web and CLI reading directly from the filesystem) toward a daemon-centric model where the daemon is the sole application boundary. The plan addresses REQ-DAB-1 through REQ-DAB-15, covering migration of the web layer's 26+ filesystem read operations, the CLI's direct git and filesystem operations, and agent toolbox architecture.

The plan identifies current gaps: web server components read from disk via shared `lib/` utilities; two web API routes perform direct writes; CLI commands operate entirely outside the daemon; and agent capabilities operate through internal callbacks rather than a public skill contract. The migration is structured as eight independently shippable phases ordered by foundational dependencies and risk: daemon read API endpoints (Phases 0-1), web layer migration (Phases 2-3), CLI migration (Phase 4), route reorganization to capability-oriented grammar (Phase 5), skill contract formalization (Phase 6), and agent skill projection (Phase 7).

The plan includes comprehensive gap analysis, phase-by-phase implementation guidance, eight open questions requiring resolution, five identified risks with mitigation strategies, and a decision log. Two requirements are already satisfied by current architecture (REQ-DAB-6 on session lifecycle, REQ-DAB-13 on internal decomposition). A standing validation rule (REQ-DAB-14) applies to every phase: confirm no new client-side authority paths or boundary bypasses were introduced.

DECISIONS AND REASONING

No decisions were made. Octavia offered three paths forward: thorough review of the plan against its spec and design doc, focus on a specific phase or concern, or alternative approach.
