  SUMMARY
  
  The session began with the Writer invoking the lore-development review-ideas skill, which surfaced one open idea from 2026-02-26 questioning whether meeting and commission data should be stored entirely in frontmatter. The Writer attempted to gather clarifying questions but was blocked by session mode restrictions, so the user provided the rationale directly. The core problem is a rendering one: notes_summary fields contain full markdown documents embedded inside YAML block scalars inside frontmatter, creating three levels of nesting. Displaying this content in a web page requires special extraction logic, which defeats the purpose of storing content as markdown in the first place. The user's position was clear: frontmatter should hold structured metadata only, never formatted text.
  
  The user defined the fix direction for both document types. For meetings, notes_summary should move to the markdown body — it is the human-facing output of the session and has no business living in a metadata block. For commissions, result_summary occupies the same role: it is the final, authoritative thing the commission produces and the content users actually care about. Both fields are presently stored as escaped strings or block scalars inside YAML, requiring custom parsing to render. Both should be the document body.
  
  The Writer confirmed the problem by reading meeting-artifact-helpers.ts, commission-artifact-helpers.ts, and an actual completed meeting artifact, which showed a full structured document (headers, horizontal rules, paragraphs, linked artifacts, follow-ups) living entirely inside a YAML notes_summary field with an empty markdown body below. The commission artifact mirrored this with result_summary. With the problem confirmed and scope defined, the Writer created a formal issue document capturing the diagnosis, impact, fix direction, and affected code paths across daemon services and lib modules.
  
  
  DECISIONS MADE
  
  The frontmatter/body split idea from 2026-02-26 was promoted to a formal issue rather than left as an open idea. The decision on where the line falls was settled in this session: structured data (status, dates, worker, timeline entries, linked artifact paths) stays in frontmatter; human-readable content (notes_summary for meetings, result_summary for commissions) moves to the markdown body. This is consistent with how markdown files are intended to work and eliminates the need for special display processing.
