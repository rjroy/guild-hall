P4 Adapter Spec Review — March 25, 2026

This session was a review of the P4 adapter spec (p4-adapter.md), a standalone Bun CLI tool that bridges Perforce workspaces with Guild Hall's git-based workflow. The spec covers two commands: init, which creates a disposable git repo from a P4 workspace with a baseline commit, and the command under review. The bulk of the meeting focused on naming and safety constraints around that second command.

The central issue raised was that the command was originally named "submit," which collides directly with p4 submit — the P4 operation that permanently commits files to the depot. The adapter command does no such thing; it only creates a shelved changelist. With P4 practitioners as the target audience, this naming created real cognitive risk. The rename to "shelve" was agreed immediately. Octavia also noted that the spec lacked an explicit prohibition on calling p4 submit, which was identified as a gap worth closing beyond just renaming.

Changes were applied to the spec in session. The command was renamed from submit to shelve across approximately 40 occurrences, including file names (shelve.ts, shelve.test.ts), CLI output strings, error messages, and the cycle description. A new requirement, REQ-P4A-36, was added to the Architecture Boundary section explicitly prohibiting the adapter from ever calling p4 submit. The requirement names the adapter's permitted P4 write surface: p4 add, p4 edit, p4 delete, p4 shelve, and p4 revert. A corresponding test case (#25) was added, p4 submit was added to the Out of Scope section, and a success criterion was added to the checklist. Five references to "p4 submit" remain in the spec, all intentional: the prohibition in REQ-P4A-36, the test case enforcing it, the Out of Scope entry, the success criterion, and one instance using "submitted" as an English verb describing what other P4 users do to the depot.

Key decision: rename the adapter's second command from "submit" to "shelve" and add REQ-P4A-36 as an explicit architectural constraint prohibiting p4 submit. Reasoning: the name must match the operation, and the prohibition should be stated plainly rather than implied by omission.

Artifacts referenced: .lore/specs/infrastructure/p4-adapter.md (modified in session).

No open items. The spec was closed out at the end of the session with no follow-up actions identified.

## Decisions

**Should the adapter's second command be called "submit" or "shelve"?**
Rename to "shelve". Add explicit safety constraint: the adapter must never call `p4 submit`.
*Reasoning: In P4 context, "submit" means `p4 submit` (permanent depot commit). The adapter only creates shelved changelists. The naming collision creates cognitive risk: someone reading the spec or CLI help could expect depot-level changes. The command should be named for what it does. Additionally, an explicit safety constraint prevents any future implementer from adding `p4 submit` support without a deliberate spec change.*
