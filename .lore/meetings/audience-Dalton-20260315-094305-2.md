MEETING NOTES: CSS `composes` Attribute

SUMMARY
Investigated why CSS Modules `composes` directives weren't functioning in the Guild Hall codebase. Found two files using the feature: MeetingRequestCard.module.css and CommitLoreButton.module.css, with 9 total composed class instances. The `composes` directive itself is valid CSS Modules syntax, but Turbopack (the Next.js bundler configured via `next dev --turbopack`) does not support it. Turbopack's Rust-based CSS Modules implementation silently ignores `composes` directives, causing buttons to receive only their specific style declarations without inheriting the composed base class properties. This created partial visual correctness that masked the underlying styling gap.

DECISIONS MADE
Decided to eliminate `composes` usage by composing class names on the component level using template literals. Removed all `composes: actionButton` and `composes: commitButton` directives from the two CSS modules. Updated 9 button elements across MeetingRequestCard.tsx and CommitLoreButton.tsx to apply both class names explicitly (e.g., `className={`${styles.actionButton} ${styles.openButton}`}`). This approach is compatible with Turbopack and maintains the same visual and functional result without relying on unsupported CSS Modules preprocessing.

ARTIFACTS PRODUCED
Retro document created at `.lore/retros/css-composes-turbopack-incompatibility.md` recording two lessons: (1) Critical lesson added to CLAUDE.md documenting Turbopack's lack of `composes` support and recommending TSX-side composition as workaround; (2) Valid lesson retained in retro noting how silent compatibility gaps can hide beneath partial visual correctness.
