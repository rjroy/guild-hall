GUILD HALL WINDOWS NATIVE SUPPORT BRAINSTORM REVIEW
Meeting with Octavia | 2026-03-24

SUMMARY

Reviewed comprehensive brainstorm on implementing Windows native support for Guild Hall. The core blocker is Unix socket dependency (limited to four files: daemon/index.ts, daemon/lib/socket.ts, lib/daemon-client.ts, lib/paths.ts). Beyond sockets, identified six non-socket POSIX assumptions requiring fixes: HOME environment variable fallback, shell dispatch (`sh -c` → `cmd.exe /c`), git long path support, line ending consistency, signal handling, and SSE idle timeout behavior.

Examined three IPC alternatives for Windows: TCP localhost (simplest, Bun-native, weaker security isolation), Named Pipes (strongest security, no Bun support, requires node:http fallback), and Windows AF_UNIX (ideal if supported, currently unverified in Bun). Recommended architecture is Option 2 (TCP fallback on Windows only) with four-phase migration path: Phase 0 (low-risk portability fixes), Phase 1 (transport abstraction), Phase 2 (verification and hardening), Phase 3 (polish and documentation).

Began systematic review of five open questions critical to finalizing the approach.

KEY ANALYSIS

Question 1 (Bun unix: support on Windows): Knowledge cutoff May 2025 shows no support; requires verification against current Bun releases via issue tracker and changelog. If recently added, eliminates ~60% of planned work.

Question 2 (node:http with socketPath on Windows): Unverified but probable via Bun's Node compat layer. Affects viability of Option 4 (node:http.createServer() server with Windows AF_UNIX socket). Requires hands-on testing, not documentation research.

Question 3 (TCP security acceptability): TCP localhost is process-visible but not network-visible. Threat model is limited (local process access only, no authentication on daemon HTTP API). Acceptable for personal dev tool but needs documented guidance for security-conscious users. Stronger isolation available via Named Pipes at cost of medium complexity.

OPEN ITEMS

Verify against current Bun releases (1.1+, as of March 2026):
- Has `Bun.serve({ unix })` been added to Windows since mid-2025?
- Does `node:http` with `socketPath` work in Bun on Windows for SSE streaming?

Complete review of remaining two open questions:
- Should transport be explicitly configurable (config.yaml setting) vs. auto-detected by platform?
- Does Claude Agent SDK sandbox mechanism work correctly on Windows, and what guarantees does it provide?

Determine security posture: Accept TCP localhost isolation with clear documentation, or defer Windows support until stronger isolation (Named Pipes) is viable?
