# Meeting Notes: CLI Agent-First Surface Spec
**Date:** 2026-04-19  
**Audience:** Octavia  
**Status:** Draft spec completed, ready for planning phase

## Summary

The meeting addressed a structural gap in the CLI's progressive discovery model. The current CLI mirrors the daemon's domain-centric grammar (`<toolbox>/<feature>/<object>/<operation>`), which produces artifacts incompatible with agent-first interaction: duplicated segments (`commission request commission list`), phase labels occupying path positions, and missing fundamental operations (`list projects`, `list active meetings`). The spec separates concerns — the daemon keeps its domain grammar for routing and worker eligibility, while the CLI owns a noun-centric surface designed for agents discovering capabilities by object type. 

The solution introduces a CLI-owned mapping between agent-friendly paths and daemon operation IDs, removing the `cliPath` presentation hint from `OperationDefinition`. This restores a clean boundary: the daemon publishes what it can do, the CLI decides how to present it. Four new daemon operations are added to complete the agent surface (`project.list`, `meeting.list` with aggregation, `issue.list`, `issue.read`), and `help` is enriched with structured, traversable output at every node — enabling a Claude session to construct its own skill definition from CLI introspection alone, with no external documentation.

## Key Decisions and Reasoning

**CLI owns the mapping.** The `cliPath` field was intended as a bridge between user words and daemon hierarchy, but it delegated presentation decisions to the daemon. When organizing principles diverge — domain-centric vs. noun-centric — the presentation hint belongs with the consumer. Removing it and housing the mapping in the CLI package restores separation of concerns and allows clean divergence without daemon carrying UX metadata.

**Four daemon leaf operations are in-scope.** Rather than spec the target surface and defer each missing leaf as a separate issue, the four operations necessary for a complete agent surface are bundled here. Larger daemon grammar restructuring (phase labels) remains out of scope; this spec addresses gaps at the operation level only.

**One acknowledged aggregation.** `meeting list` routes to both `meeting.request.meeting.list` and `meeting.session.meeting.list`, collapsing phase-separated operations into a single discoverable command with a state filter. This is the exception; the default remains one CLI leaf → one daemon call. Future aggregations require the same explicit call-out.

**Help-driven skill construction.** REQ-CLI-AGENT-19 and REQ-CLI-AGENT-20 establish that a Claude session should build a complete skill definition from recursive `help --json` output without supplementary sources. This makes the CLI self-documenting at the level of granularity agents need.
