Meeting with Octavia — 2026-03-25

The session focused on a critical performance problem in the p4-adapter integration with Guild Hall: git worktree creation against the EOS SDK workspace (148,973 files, 1.88GB at minimum) takes approximately 3 minutes per worktree. Because every commission and meeting creates a worktree, and Ronald runs upwards of 60 commissions per day across brainstorm, design, spec, plan, review, and fix cycles, the cumulative cost is roughly 3 hours of dead time per day before any work begins. The 20-minute p4-adapter init is a separate problem but is a one-time-per-cycle cost; the per-commission worktree creation is the usability wall.

The root cause was identified in workspace.ts: git worktree add fully materializes all 148,973 files to disk before sparse checkout prunes them. Using --no-checkout with sparse checkout only helps for lore-scoped commissions; Dalton (the developer worker) needs the full tree accessible for context and build, so sparse checkout is not a viable general solution. A prior research document on Windows ProjFS was revisited and found to have asked the wrong question. ProjFS cannot replace worktrees, but it can make worktree population lazy: files appear instantly as virtual placeholders and hydrate from the git object store only when actually read. Even a development commission reading 500 files hydrates under 0.5% of the tree. The grep/ripgrep hydration concern is real but bounded — ripgrep hydrates files it opens, but type-filtered searches touch a fraction of the workspace, and git grep (which reads blobs directly from the object store) avoids hydration entirely for read-only search.

No final decisions were made on architecture or implementation. The brainstorm document was written and saved to capture the full analysis, including two architectural options (ProjFS as an overlay on real git worktrees using --no-checkout versus ProjFS projecting directly from the object store without a git working tree), the C# managed API + LibGit2Sharp provider design, daemon lifecycle integration as a managed subprocess, and the platform strategy of transparent fallback on non-Windows. Open questions left in the document include git status behavior on 149,973 ProjFS placeholders without FSMonitor, Windows Defender interaction with placeholders, concurrent provider processes sharing the object store, and whether a git grep MCP tool is worth building or scoped ripgrep usage is sufficient. A separate quick-win was called out: the p4-adapter init.ts attrib and git add steps should be scoped to in-scope files only, independent of ProjFS work, and would reduce the 20-minute init time on its own.

Artifact produced: .lore/brainstorm/projfs-lazy-worktree-population.md

Open items: Validate git status behavior on ProjFS placeholders (may require FSMonitor), prototype the C# ProjFS provider against the EOS SDK object store, assess Windows Defender impact, and decide whether git grep tooling is in scope. The 20-minute init optimization in p4-adapter/init.ts is a separable code change that can proceed independently.
