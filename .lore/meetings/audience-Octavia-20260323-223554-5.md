**GUILD HALL MEETING NOTES**
March 24, 2026 | Octavia Audience

---

**INVESTIGATION SUMMARY**

The team investigated why `/prep-plan` skill workers were defaulting plan artifacts to `status: active` instead of the correct `status: draft`. This created inconsistent artifact metadata across the system, as plans should begin life in draft status pending user approval, not active status (which is reserved for scheduled commissions and research documents).

The root cause was identified in the frontmatter schema documentation at `shared/frontmatter-schema.md`. While the schema file defines correct status values for every document type (specs, designs, research, brainstorms, retros, etc.) with concrete examples, it conspicuously omits an example for plans. Workers implementing `/prep-plan` therefore had no reference and inferred behavior from adjacent document types, settling on `active` as a plausible default.

The fix is straightforward: add a plan example to the Examples section of the frontmatter schema (around line 109), demonstrating that new plans initialize with `status: draft`, transition to `approved` upon user acceptance, and move to `executed` after implementation completes. This single-source-of-truth pattern ensures all future `/prep-plan` invocations will produce correctly-scoped artifacts.

---

**KEY DECISION**

Update `shared/frontmatter-schema.md` to include a plan example between the existing design and research examples. This closes the documentation gap that caused inconsistent worker behavior without requiring changes to worker logic itselfâ€”the schema is the canonical reference workers delegate to.
