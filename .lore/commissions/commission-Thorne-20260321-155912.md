## Review: Triggered Commissions Core (Steps 1-4)

### Verdict

Clean implementation. Four findings, two at WARN level and two informational. No blockers.

### Finding 1 — WARN: `triggered_by` YAML values are not escaped

**File:** `daemon/services/commission/orchestrator.ts:1351-1352`

The `triggered_by` block interpolates `sourceId` and `triggerArtifact` directly into the YAML template without escaping:

```typescript
const triggeredByBlock = options?.sourceTrigger
  ? `\ntriggered_by:\n  source_id: ${options.sourceTrigger.sourceId}\n  trigger_artifact: ${options.sourceTrigger.triggerArtifact}\n  depth: ${options.sourceTrigger.depth}`
  : "";
```

Both `sourceId` and `triggerArtifact` are commission IDs (format: `commission-WorkerName-YYYYMMDD-HHMMSS`), so they will never contain YAML-breaking characters in practice. But the `source_schedule` field on line 1348 follows the same pattern (unescaped), establishing that this is a conscious convention for ID-shaped values.

**Impact:** Theoretical. Commission IDs are generated by `formatCommissionId()` which produces colon-free, quote-free strings. If a future caller passes arbitrary strings through `sourceTrigger`, the YAML would break. The `source_schedule` field has the same exposure.

**Recommendation:** Acknowledge or quote. If the convention is "IDs don't need escaping," fine. If you want defense in depth, wrap values in `escapeYamlValue()` as done for `title` and `prompt`. Either way, this is the caller's domain (the trigger evaluator in Step 5), and it controls the values passed.

---

### Finding 2 — WARN: `readTriggerMetadata` uses gray-matter for the `match` sub-block, which coerces YAML types

**File:** `daemon/services/commission/record.ts:378-400`

`readTriggerMetadata` parses via gray-matter, then casts `trigger.match` directly to `TriggerBlock["match"]` (which is `EventMatchRule`). The `EventMatchRule.fields` property is `Record<string, string>`, but gray-matter's YAML parser will coerce values like `"true"` → `true` (boolean), `"123"` → `123` (number). If a trigger's `fields` block contains `status: true` or `count: 5`, the parsed result would have non-string values in `fields`, violating the `Record<string, string>` contract.

The `EventRouter.matchesRule()` calls `micromatch.isMatch(String(actual), pattern)`, so it will `String()` the event field value before matching. But the *pattern* (from `fields`) gets passed directly. If gray-matter coerces the pattern to a boolean, `micromatch.isMatch("completed", true as any)` would produce unpredictable behavior.

**Evidence:** The test fixture at `record.test.ts:1072` uses `status: completed`, which gray-matter returns as a string. But a fixture with `enabled: true` would return a boolean.

**Impact:** Any trigger matching on a field whose expected value looks like a YAML primitive (`true`, `false`, `null`, a bare number) will silently mismatch. This is the same gray-matter type coercion issue documented in CLAUDE.md for other contexts.

**Recommendation:** After parsing, coerce `fields` values to strings: `Object.fromEntries(Object.entries(match.fields).map(([k, v]) => [k, String(v)]))`. Or quote field values in the YAML spec ("values in `fields` must be quoted strings").

### Finding 3 — INFO: `readTriggerMetadata` handles `Date` objects from gray-matter for `last_triggered`

**File:** `daemon/services/commission/record.ts:395-397`

```typescript
last_triggered: trigger.last_triggered === null || trigger.last_triggered === undefined
  ? null : trigger.last_triggered instanceof Date
    ? trigger.last_triggered.toISOString() : String(trigger.last_triggered),
```

This correctly handles gray-matter's YAML date coercion (ISO date strings get parsed into `Date` objects). Good defensive code. No action needed.

### Finding 4 — INFO: `expandTemplate` regex accepts dotted field names but no event has dotted keys

**File:** `daemon/services/trigger-evaluator.ts:22`

The regex `/\{\{(\w+(?:\.\w+)*)\}\}/g` matches both `{{status}}` and `{{foo.bar}}`. The function then does a flat lookup on the event object: `record[fieldName]`, so `{{foo.bar}}` looks for a key literally named `"foo.bar"`, which won't exist. This is exactly the behavior the plan specifies (REQ-TRIG-11: "No nested access. `{{foo.bar}}` is treated as a field named 'foo.bar'"). The regex is broader than necessary (it could be just `\w+`), but it's harmless because the lookup behavior is correct regardless.
