We need to support both layouts concurrently.\n\n## Discovery Input\n\nRead **`.lore/issues/lore-directory-restructure-discovery.md`** first. Verity's discovery report catalogs every assumption point (~30 call sites across path resolution, listing, writing, type classification, UI, CLI, toolboxes, prompts, and tests), grouped by category with file:line references. Treat it as the input artifact — the spec must close every open question it raises and cover every site it lists.\n\n## User Decisions (Already Made)\n\nThese are settled. Do not re-open them:\n\n1. **Coexistence (Reading B).** `.lore/specs/foo.md` and `.lore/work/specs/bar.md` may exist in the same project at the same time. Both surface as Specs. UI merges them. Reason: migrations across projects will take time; existing work must keep functioning.\n\n2. **Peel.** `work/` is a layout prefix that gets peeled before type classification. `.lore/work/specs/foo.md` classifies as type `Spec`, not as `{layoutGroup: \"work\", type: \"Spec\"}`. Single-axis classification, drop-in compatible with current consumers.\n\n3. **Commissions and meetings move under `work/`.** New artifacts go to `.lore/work/commissions/<id>.md` and `.lore/work/meetings/<id>.md`. Old artifacts at `.lore/commissions/` and `.lore/meetings/` continue to be readable (coexistence applies to activity artifacts too).\n\n4. **Add `learned` to `TYPE_LABELS`.** Alongside `reference`.\n\n5. **Root-level lore files stay at root.** `heartbeat.md`, `lore-config.md`, `lore-agents.md`, `vision.md` are non-typed lore. Document the rule.\n\n6. **`.lore/generated/` stays flat.** Generated images are not typed work artifacts.\n\n## Your Task\n\nDraft a spec at `.lore/specs/infrastructure/lore-directory-restructure.md` that:\n\n### Required Sections\n\n- **Status:** `draft`\n- **Goals** — one paragraph per goal (coexistence, peel-based classification, write-new-to-`work/`)\n- **Non-goals** — explicitly out of scope (migrating existing artifacts, tooling to move artifacts between layouts, removing flat layout)\n- **Requirements** — numbered REQ-LDR-* covering:\n  - Path resolution: how `commissionArtifactPath`, write paths in orchestrator/toolbox/triage/record evolve. New writes go under `work/`. Reads accept both.\n  - Type classification: `artifactTypeSegment` peels a leading `work/` segment before extracting type. `learned` joins `reference` in `TYPE_LABELS`. Define behavior for unknown segments under `work/` (passthrough? null?).\n  - Discovery/listing: `scanArtifacts` walks both layouts. `recentArtifacts` filter migrates from raw-segment matches to canonical-type matches.\n  - Artifact tree/grouping: `groupKey` and `buildArtifactTree` peel `work/` so the tree shows Specs/Plans/etc. as top-level groups, not \"Work\". Commissions and meetings stay grouped as themselves regardless of layout.\n  - Workspace-issue routes: list reads both `.lore/issues/` and `.lore/work/issues/`. Create writes to `.lore/work/issues/` (the new home).\n  - Artifacts route prefix detection (`apps/daemon/routes/artifacts.ts:111-121`): handle `work/meetings/`, `work/commissions/` prefixes alongside flat `meetings/`, `commissions/`.\n  - Squash-merge auto-resolution (`apps/daemon/lib/git.ts:617`, `services/workspace.ts:284`): confirm `.lore/work/*` inherits the same auto-theirs policy. Already does because the prefix match is `.lore/`.\n  - Sparse checkout: `[\".lore/\"]` already covers `work/`. Confirm and document.\n  - LLM-facing strings: briefing prompt, tool descriptions, operation descriptions, worker postures, skill files. List the files that need updating with new layout language. The spec must enumerate them — do not defer to \"discovery.\"\n  - Tests: parallel coverage for the new layout. Existing flat-layout tests stay; new tests target peel behavior and dual-write behavior.\n\n### Open Questions Section\n\nEmpty unless something genuinely cannot be answered without user input. Do not fabricate questions. Do not write speculative \"what if\" entries.\n\n### Acceptance Criteria\n\nA short, checkable list a reviewer can run against an implementation.\n\n### Reference Section\n\nLink the discovery report (`.lore/issues/lore-directory-restructure-discovery.md`) and any specs the reader needs for context (the daemon-application-boundary spec is relevant for write-boundary rules).\n\n## Style Notes\n\n- Project memory section \"Process Decisions\" — read for spec conventions.\n- The user has 20 years of experience and prefers precise, no-hedging prose. No em-dashes (use commas, parens, or restructure).\n- Use the existing spec format. Reference `.lore/specs/infrastructure/daemon-application-boundary.md` for shape.\n- This is a coexistence spec, not a migration spec. Do not propose tools to move artifacts.\n\nReport back with the spec path and a brief summary of the requirements you wrote."
