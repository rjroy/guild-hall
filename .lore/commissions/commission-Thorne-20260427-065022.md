## Findings (severity-ordered, exhaustive)

### 1. MINOR — `workspace.issue` feature-level description string still names the legacy path only
- **Where:** `apps/daemon/routes/workspace-issue.ts:320`
- **What:** `descriptions: Record<string, string> = { "workspace.issue": "Create and manage issues in .lore/issues/" }`
- **Why it matters:** REQ-LDR-30 requires "operation `description` and `sideEffects` strings reference `.lore/work/issues/` as the write target. Read-side descriptions note that both `.lore/issues/` and `.lore/work/issues/` are scanned." The per-operation descriptions at lines 266/268/285/303 are correct; the rollup feature description at line 320 was missed. This is an LLM-facing string consumed by CLI help/agent surfaces.
- **Fix:** `"workspace.issue": "Create and manage issues. Writes target .lore/work/issues/; reads merge .lore/work/issues/ and the flat-layout .lore/issues/."`

### 2. MINOR — `MeetingList.tsx` bare-filename fallback prepends legacy `meetings/` instead of `work/meetings/`
- **Where:** `apps/web/components/project/MeetingList.tsx:80-85`
- **What:** When `peeled` does not start with `meetings/`, the code does `\`meetings/${relativePath}\`` and the comment justifies this as "prepend the legacy prefix to preserve existing callers."
- **Why it matters:** REQ-LDR-19 spec text: "Any logic that prepends `meetings/` to a meeting filename targets the new `work/meetings/` location for newly-created meetings while still accepting flat paths read back from disk." The current branch is the opposite of what the spec calls for. Likely a dead branch in practice — every observed caller passes a full relativePath read from disk — but the literal contradiction with REQ-LDR-19 stands. Read-side resolvers handle either layout, so an emitted flat path would still resolve via dual-read; the user-visible impact is muted.
- **Fix:** Either change the prefix to `work/meetings/` (matching the spec), or remove the bare-filename branch entirely if no caller exercises it.

### 3. OBSERVATION — `apps/cli/migrate-content-to-body.ts` line 36 hard-codes `.lore/commissions/` with no dual-layout awareness
- **Where:** `apps/cli/migrate-content-to-body.ts:36`: `const commissionsDir = path.join(project.path, ".lore", "commissions");`
- **What:** REQ-LDR-33 explicitly lists this file under "comment references". Phase 4 declared "nothing to update". The file has no comments referencing the path; the path is operational code. The migration tool never scans `work/commissions/`.
- **Why it matters:** This is a one-time migration utility that lifted `result_summary` from frontmatter to body for commissions written before that change. Commissions newly written to `work/commissions/` would not have legacy `result_summary` frontmatter, so the legacy-only scan is functionally correct. But REQ-LDR-33 names the file and Phase 4 made no change. Spec drift either way: either the spec should not have named this file, or the file should at least scan both layouts.
- **Fix:** Either remove `apps/cli/migrate-content-to-body.ts` from REQ-LDR-33, or extend the migration tool to iterate `[work/commissions, commissions]`.

**Is the workspace.issue feature-level description string an LDR-30 finding?**
Yes — minor. Line 320 of apps/daemon/routes/workspace-issue.ts maps "workspace.issue" → "Create and manage issues in .lore/issues/". This is an LLM-facing feature description that still names only the legacy path.

**Does MeetingList.tsx bare-filename fallback violate REQ-LDR-19?**
Yes — minor. Lines 80-85 prepend the legacy "meetings/" prefix when the filename has no prefix; the comment explicitly justifies this as "preserve existing callers". REQ-LDR-19 says new prepends should target work/meetings/.
*Reasoning: REQ-LDR-19: "Any logic that prepends `meetings/` to a meeting filename targets the new `work/meetings/` location for newly-created meetings while still accepting flat paths read back from disk." The current fallback prepends the legacy path. Likely a dead branch in practice (callers pass full relativePath from disk), but it contradicts spec text. Minor severity because read-side resolvers handle either layout, so a path constructed with the legacy prefix would still resolve via dual-read. Spec drift: REQ-LDR-33 should be amended to remove this file or to call out it intentionally targets only the legacy layout.*
